# Closed-Loop Development Migration Skill

当用户需要将闭环研发方法论迁移到新项目时使用此 skill。
适用于任何需要长期迭代研发、有仿真/测试环境、需要视频或视觉反馈的项目。

---

## 前置条件

1. 确认项目可构建/可运行
2. 确认有可编程的反馈通道（视频录制、指标采集、日志等）
3. 确认有可用的多模态大模型 API（如 mimo-v2.5、GPT-4o 等）

---

## 迁移流程

### Phase 1: 项目分析（30min）

#### 1.1 架构分析

读取项目核心文件，识别：
- 项目的入口和启动流程
- 核心组件之间的依赖关系
- 数据/信号的流动方向

输出：信号链分层图（文本形式），通常 3-5 层：

```
第一层：调度/任务层
  │ FSM、任务队列、状态管理
  │ 若失败 → 查调度逻辑 / 超时配置
  ▼
第二层：规划/算法层
  │ 路径规划、运动规划、决策算法
  │ 若失败 → 查算法实现 / 参数配置
  ▼
第三层：控制/执行层
  │ PD控制、电机驱动、执行器
  │ 若抖 → 查控制增益 / 执行器参数
  ▼
第四层：物理/仿真层
  │ 物理引擎、传感器模拟、碰撞
  │ 若异常 → 查物理参数 / 场景配置
```

#### 1.2 反馈通道识别

列出项目所有可用的反馈信号：

| 反馈类型 | 采集方式 | 自动化程度 |
|---------|---------|----------|
| 视频/图像 | 录制/截图 | 高 |
| 数值指标 | 日志/数据库 | 高 |
| 状态转换 | 事件日志 | 高 |
| 人工观察 | 目视 | 低 |

#### 1.3 环境能力评估

确认以下能力：
- [ ] 仿真/测试环境是否支持视频录制？
- [ ] 是否有现成的指标采集机制？
- [ ] 控制台/调度系统是否可编程？
- [ ] 是否有 headless 运行模式？

---

### Phase 2: 方法论适配（2-4h）

#### 2.1 创建 AGENT.md

在项目根目录创建 `AGENT.md`，包含：

```markdown
# <项目名> 闭环研发方法论

## 一、核心原则
（写入规则：只写可复用原则，不写流水账）

## 二、闭环机制
- 机制 1：PR 驱动的渐进式开发
- 机制 2：N 层信号分离（根据项目架构）
- 机制 3：视频/反馈第一性原则
- 机制 4：多模态 AI 审查（子 Agent 隔离）
- 机制 5：分层分析器矩阵
- 机制 6：长期记忆（本文件）
- 机制 7：Git 精确追溯

## 三、PR 里程碑
（5-10 个 PR，每个有明确门控条件）

## 四、参数经验库
（随实验迭代更新）

## 五、已知问题
（随实验迭代更新）
```

#### 2.2 构建评估脚本

根据项目类型创建 `scripts/` 目录，包含：

**run_eval_episode.py** — 一键运行评估
- 启动仿真/测试环境
- 执行目标任务
- 录制视频（如适用）
- 采集指标
- 输出结构化日志

**record_video.py** — 视频录制
- 根据项目类型选择录制方案：
  - MuJoCo: native rendering (EGL headless)
  - Web: Playwright 截图/录屏
  - 游戏: replay 系统 / OBS
- 多视角录制（全局 + 局部）

**analyze_episode.py** — 数值分析器
- 根据信号链层数创建对应分析维度
- 输出 0-100 综合得分
- 输出各维度子得分

#### 2.3 定义打分准则

创建 `scripts/scoring_rubric.md`，包含：

- 5-10 项结构化检查（总分 100）
- 每项有：观察要点、评分标准、扣分规则
- 总体判定：PASS (80-100) / HIGH_RISK (60-79) / FAIL (0-59)
- 异常扣分项（摔倒、卡死、崩溃等）

#### 2.4 构建视频审查子 Agent

创建 `scripts/video_review.py`：
- 调用多模态大模型 API 审查视频/图像
- 独立环境，不共享主 Agent 上下文
- 输出结构化 JSON 报告
- 默认使用 Codex 内置的 mimo-v2.5 模型

---

### Phase 3: 验证与迭代（1-2h）

1. 运行一次完整闭环验证
2. 确认所有脚本可运行
3. 确认视频审查子 Agent 可调用
4. 确认 AGENT.md 可被新 session 读取并理解
5. 向用户报告迁移完成，请求确认

---

## PR 里程碑设计原则

1. 每个 PR 只解决一个核心问题
2. PR 之间有严格的依赖关系
3. 每个 PR 有明确的门控条件（可自动化验证）
4. 每个 PR 验收时必须确认前一个 PR 的产物仍然可用
5. PR 数量控制在 5-10 个

## 闭环数据流

```
用户给出长期研发目标
    │
    ▼
AI Agent 自主执行：
    a. 写代码 / 改配置
    b. 运行评估 episode
    c. 运行分析器
    d. 视频审查（子 Agent）
    e. 综合判断 → PASS / FAIL / HIGH_RISK
    f. 更新 AGENT.md
    g. Git 提交
    │
    ▼
下一轮迭代
```

## 关键原则

1. **视频第一性**：AI agent 必须直接"看"视频，不能只查日志
2. **三层证据交叉**：数值分析 + 视频审查 + 关键帧目视
3. **子 Agent 隔离**：视频审查必须在独立环境中运行
4. **长期记忆**：AGENT.md 只写可复用原则，不写流水账
5. **精确追溯**：每个实验 run 记录 git commit、配置、环境
